# Архитектура RAG системы

## Обзор

Система построена по принципу разделения ответственности:
- **Эндпоинты** (`rag.py`) - только HTTP обработка и валидация
- **Сервисы** (`services/`) - бизнес-логика и оркестрация пайплайна
- **Классы** (`Handling/`, `Databases/`, `LLM/`) - детальная реализация операций

## Структура проекта

```
ai-tutor-rag/
├── rag.py                    # FastAPI эндпоинты (только HTTP слой)
├── services/                 # Бизнес-логика
│   ├── __init__.py
│   ├── load_service.py       # Сервис загрузки документов
│   └── query_service.py      # Сервис обработки запросов
├── Handling/                 # Обработка данных
│   ├── Chunker.py           # Разбиение текста на чанки
│   └── Embedder.py         # Векторизация текста
├── Databases/               # Работа с базами данных
│   ├── QInteracter.py      # Взаимодействие с Qdrant
│   └── NeoInteracter.py    # Взаимодействие с Neo4j
└── LLM/                     # Работа с языковыми моделями
    └── LLMAnswer.py        # Генерация ответов
```

## Пайплайн загрузки документов

**Эндпоинт:** `POST /load`

**Поток выполнения:**

1. **Эндпоинт** (`rag.py`) - валидация запроса
2. **LoadService** (`services/load_service.py`) - оркестрация процесса:
   - `process_file()` - обработка одного файла
   - `_create_chunks()` - разбиение на чанки через `Chunker`
   - `_vectorize_chunks()` - векторизация через `Embedder`
   - `_load_to_qdrant()` - сохранение в Qdrant через `QInteracter`
   - `_load_to_neo4j()` - создание графа через `NeoInteracter`

**Детали реализации скрыты в классах:**
- `Chunker.make_chunks_from_text()` - разбиение текста
- `Embedder.embed()` - создание векторных представлений
- `QInteracter.load_in_qdrant()` - сохранение в векторную БД
- `NeoInteracter.create_graph()` - создание графа знаний

## Пайплайн обработки запросов

**Эндпоинт:** `POST /query`

**Поток выполнения:**

1. **Эндпоинт** (`rag.py`) - валидация запроса
2. **QueryService** (`services/query_service.py`) - оркестрация процесса:
   - `process_query()` - основной метод обработки
   - `_rewrite_question()` - перефразирование через `LLM`
   - `_extract_aspects()` - извлечение аспектов через `QInteracter`
   - `_search_relevant_chunks()` - поиск в Qdrant через `QInteracter`
   - `_build_graph_context()` - построение контекста через `NeoInteracter`
   - `_generate_answer()` - генерация ответа через `LLM`

**Детали реализации скрыты в классах:**
- `LLM.rewrite_question_from_dialogue()` - перефразирование вопроса
- `QInteracter.extract_aspects_from_question()` - извлечение аспектов
- `QInteracter.dense_search()` - семантический поиск
- `NeoInteracter.graph_context_from_chunks()` - построение графового контекста
- `LLM.answer_with_graph()` - генерация финального ответа

## Принципы архитектуры

### 1. Разделение ответственности
- **Эндпоинты** - только HTTP слой, валидация, форматирование ответов
- **Сервисы** - бизнес-логика, оркестрация пайплайна
- **Классы** - детальная реализация конкретных операций

### 2. Сокрытие реализации
Программист, работающий с системой, видит:
- Понятный пайплайн в сервисах
- Высокоуровневые методы с понятными названиями
- Детали реализации скрыты в классах

### 3. Легкость навигации
- Начав с эндпоинта, легко проследить весь пайплайн
- Каждый уровень абстракции понятен и самодостаточен
- Можно углубиться в детали реализации при необходимости

## Пример чтения кода

**Для понимания пайплайна загрузки:**
1. Откройте `rag.py` → эндпоинт `/load`
2. Перейдите в `services/load_service.py` → метод `process_files()`
3. Изучите методы `_create_chunks()`, `_vectorize_chunks()`, и т.д.
4. При необходимости углубитесь в классы (`Chunker`, `Embedder`, и т.д.)

**Для понимания пайплайна запросов:**
1. Откройте `rag.py` → эндпоинт `/query`
2. Перейдите в `services/query_service.py` → метод `process_query()`
3. Изучите методы `_rewrite_question()`, `_extract_aspects()`, и т.д.
4. При необходимости углубитесь в классы (`LLM`, `QInteracter`, и т.д.)

## Компоненты системы

### Handling/
- **Chunker** - разбиение текста на чанки с перекрытием
- **Embedder** - векторизация текста через модель BAAI/bge-m3

### Databases/
- **QInteracter** - работа с векторной БД Qdrant (семантический поиск)
- **NeoInteracter** - работа с графовой БД Neo4j (граф знаний)

### LLM/
- **LLM** - работа с языковыми моделями (перефразирование, генерация ответов)

## Преимущества такой архитектуры

1. **Понятность** - легко понять общий пайплайн работы
2. **Масштабируемость** - легко добавлять новые этапы обработки
3. **Тестируемость** - каждый компонент можно тестировать отдельно
4. **Поддерживаемость** - изменения в одном слое не затрагивают другие
5. **Читаемость** - код самодокументируется через структуру и комментарии
